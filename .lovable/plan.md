

# خطة إعادة هيكلة السائقين حسب الطبيعة الفعلية

## المشكلة الجوهرية

الكود الحالي يعامل السائق الحر المؤجر (Hired) كأنه لديه حساب كامل بلوحة تحكم وقوائم جانبية (سوق شحنات، عروض، عقود، محفظة، تحليلات) — وهذا **خاطئ تماماً**. السائق المؤجر ليس له حساب أصلاً، فقط يستلم **رابط لمرة واحدة** لتنفيذ شحنة واحدة فقط ودوره ينتهي.

---

## الفرق الجوهري بين الثلاثة (حسب رؤيتك)

```text
┌──────────────────┬────────────────────┬──────────────────────┬─────────────────────────┐
│                  │ سائق تابع (Company)│ سائق مؤجر (Hired)    │ سائق مستقل (Independent)│
├──────────────────┼────────────────────┼──────────────────────┼─────────────────────────┤
│ العلاقة بالجهة   │ موظف دائم          │ خارجي - لا علاقة      │ مستقل - يسجل نفسه       │
│ له حساب؟         │ نعم + كلمة مرور    │ لا — رابط مؤقت فقط   │ نعم — تسجيل ذاتي        │
│ لوحة تحكم؟       │ نعم — كاملة        │ لا — صفحة واحدة فقط  │ نعم — تطبيق مثل Uber    │
│ الصلاحيات        │ كاملة (شحنات+خرائط)│ صفر — الشحنة فقط     │ استقبال طلبات + تسعير   │
│ آلية الإسناد     │ الجهة مباشرة       │ رابط خارجي من الجهة  │ عرض/طلب (Uber model)    │
│ مدة الارتباط     │ دائم               │ مرة واحدة — ينتهي    │ حسب الطلب               │
│ يرى حسابات الجهة│ نعم               │ لا أبداً              │ لا أبداً                │
└──────────────────┴────────────────────┴──────────────────────┴─────────────────────────┘
```

## خطة التنفيذ (3 مراحل)

### المرحلة 1: إعادة تعريف السائق المؤجر (Hired) — الأهم

**الوضع الحالي الخاطئ**: لديه لوحة تحكم كاملة مع 8+ عناصر قائمة
**الوضع الصحيح**: لا يملك حساب. الجهة ترسل له رابط خارجي لمرة واحدة

1. **إنشاء نظام "الرابط المؤقت" (One-Time Mission Link)**:
   - الجهة الناقلة تنشئ "مهمة خارجية" وتحدد تفاصيل الشحنة
   - النظام يولّد رابط فريد (مثل `/mission/:token`) صالح لمرة واحدة
   - السائق المؤجر يفتح الرابط → يرى صفحة واحدة فقط فيها تفاصيل المهمة
   - يدخل بيانات التنفيذ (الكمية، المسار، التأكيد) → ينتهي دوره

2. **حذف قائمة السائق المؤجر من `useDriverMenu.ts`**: لم يعد بحاجة لقائمة جانبية لأنه لا يملك لوحة تحكم

3. **تعديل `driver-types.ts`**: تحديث الوصف ليعكس أنه "سائق خارجي يستلم رابط لمرة واحدة"

### المرحلة 2: تعزيز السائق المستقل (Independent) — نموذج Uber/InDriver

1. **تطوير "سوق الشحنات" كتطبيق مصغر**:
   - واجهة تشبه Uber/InDriver: عرض الشحنات المتاحة حوله جغرافياً
   - نظام العرض والطلب: الجهة تعرض سعر + السائق يقبل أو يرفض أو يفاوض
   - السائق يحدد أجرته + الجهة تحدد ميزانيتها → تطابق ذكي

2. **تحسين لوحة التحكم المستقل**:
   - Dashboard يشبه تطبيقات التوصيل (خريطة + طلبات واردة + أرباح)
   - إحصائيات الأداء والتقييم
   - المحفظة والأرباح

### المرحلة 3: تثبيت السائق التابع (Company)

1. **تأكيد الصلاحيات الكاملة**: إنشاء شحنات، إغلاقها، الخرائط، المواقع، البيانات
2. **ربط ببيانات الجهة**: الرقم القومي، بيانات التعيين، الرخصة
3. **التأكد من أن السائق التابع يظهر في هيكل الجهة** كموظف فعلي
